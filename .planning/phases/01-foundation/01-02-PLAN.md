<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client app → Firebase Auth/Firestore | Auth and profile data cross the network only in live mode |
| app → browser localStorage | Mock-mode profile/journal/chat data persists client-side |
| build env → client bundle | `import.meta.env.VITE_*` is inlined into shipped JS |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Denial of Service | localStorage writes (dbService lsSet) | mitigate | Wrap every setItem in try/catch; on QuotaExceededError console.warn and return without throwing (Pitfall 3). |
| T-01-05 | Information Disclosure | localStorage in shared browsers | accept | Namespaced `zenstudy:` keys; only non-sensitive profile/journal data stored; no secrets ever written to localStorage (RESEARCH Security Domain). |
| T-01-06 | Tampering | Firestore partial writes (updateUser) | mitigate | Use `setDoc(..., { merge: true })` so streak/zenPoints/redeemedItems fields are never clobbered (Pitfall 4). |
| T-01-07 | Information Disclosure | Gemini API key in client | mitigate | Phase 1 never sends the key; live branch guarded behind GEMINI_MOCK_MODE; key handling deferred to Phase 4 Settings panel; never hardcoded (D-10). |
| T-01-08 | Denial of Service | anonymous auth quota (live mode) | accept | Default mock mode bypasses Firebase entirely for examiners (Pitfall 5); rate-limiting is Firebase's responsibility in live mode. |
| T-01-SC | Tampering | npm installs (firebase, @google/generative-ai) | mitigate | RESEARCH Package Legitimacy Audit approved both (official Google/Firebase orgs, no postinstall); no unresolved [ASSUMED]/[SUS] packages, so no blocking checkpoint required. |
</threat_model>
